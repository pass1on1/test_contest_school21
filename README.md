# Модуль интеллектуальной обработки документов (тестовое задание)

AI-агент проверки целевого использования льготных сельхоз-кредитов: извлечение
полей из документов, классификация типа документа и проверка соответствия
предмета оплаты условиям льготной программы.

## 1. Как запустить

```bash
pip install -r requirements.txt
pytest
```

Всё выполняется офлайн, без ключей API.

## 2. Технологии

- **Python 3.11+, стандартная библиотека (`re`, `datetime`) — без сторонних NLP/LLM-зависимостей.**
  Задача — извлечение полей и классификация по формализуемым правилам
  (форматы сумм/дат конечны, категории документов и типов предмета оплаты —
  фиксированный список), поэтому регулярные выражения и словари ключевых слов
  дают предсказуемый, объяснимый и быстро тестируемый результат без
  необходимости в модели или API-ключе.
- **pytest** — юнит-тесты, единственная внешняя зависимость.
- LLM/LangChain для `check_subject()` сознательно не подключались — см. раздел
  «Компромиссы».

## 3. Архитектура

```
document_agent/
    extract.py        # extract(text) -> dict: amount, date, inn, contractor, subject
    classify.py        # classify(text) -> (doc_type, confidence)
    check_subject.py   # check_subject(subject) -> (matches, confidence, reason)
dataset/                # тестовые документы (из репозитория задания)
tests/                  # pytest-тесты
scripts/
    run_extract_report.py   # таблица extract()/classify() по всем файлам dataset/
    run_subject_report.py   # таблица check_subject() по subjects_test.txt
conftest.py             # добавляет корень репозитория в sys.path для тестов
```

**`extract()`**: суммы ищутся по regex рядом со словами-валютами
(`руб/₽/RUB`), нормализуются с учётом любой комбинации разделителей
тысяч/десятичных (пробел, запятая, точка); среди нескольких найденных сумм
выбирается наиболее вероятный итог по контексту (слова "итого", "составляет"
получают бонус, "НДС"/"в том числе НДС" — штраф, "итого с НДС" — явный
приоритет над "итого без НДС"). Если в тексте сумма указана только словами
("девятьсот тысяч рублей") — работает отдельный парсер русских числительных.
Даты ищутся по трём форматам (`ДД.ММ.ГГГГ`, `ДД/ММ/ГГГГ` или `ММ/ДД/ГГГГ`,
"1 марта 2025 г.") и берётся **первая по тексту** распознанная дата — как
правило, это дата самого документа. ИНН и контрагент — по меткам-подписям и
паттернам организационно-правовых форм (ООО/АО/ИП).

**`classify()`**: взвешенная сумма совпадений ключевых слов по 4 типам
документов, с двойным весом для совпадений в заголовке (первые 120 символов)
— заголовок документа надёжнее ссылок на него в чужом тексте. Если разрыв
между первым и вторым местом меньше порога — возвращается `unknown`.

**`check_subject()`**: список разрешённых сельхоз-категорий (агрохимия,
семена, техника, топливо, запчасти, полевые работы, страхование урожая)
против списка типичных "не сельхоз" категорий (аренда офиса, юридические
услуги, маркетинг/IT, консультации и т.д.), сопоставление по ключевым словам.

## 4. Компромиссы

- **`check_subject()` без LLM.** В задании предложены оба варианта (rule-based
  и LLM-промпт с fallback на rule-based). В условиях дедлайна выбран только
  rule-based вариант — он и есть требуемый offline-fallback, поэтому
  реализация LLM-пути не добавляла бы новой функциональности, только риск
  падения из-за отсутствия ключа. При наличии времени LLM-путь стоило бы
  добавить для предметов оплаты со свободным текстом, которые не попадают ни
  в одну словарную категорию.
- **`classify()`: порог уверенности = 2** (разница весов топ-1 и топ-2
  категории). Подобран на 7 файлах датасета — при пороге 2 все размеченные
  документы классифицируются верно, а испорченный OCR-документ
  (`scan_ocr_001.txt`, где ключевые слова не распознаются из-за подмены
  букв цифрами) уходит в `unknown`, а не в случайную категорию. При
  уменьшении порога (напр. до 0) риск ложной уверенной классификации по
  единичному случайному совпадению растёт; при увеличении (напр. до 5) в
  `unknown` начинают попадать и вполне уверенно определяемые документы,
  снижая полноту.
- **Дата документа = первая распознанная в тексте, а не дата "к оплате"/
  "не позднее".** На 4 из 5 файлов датасета это совпадает с ожидаемым
  результатом (дата в шапке документа). Исключение — `invoice_002.txt`: в
  тексте есть и дата счёта (15 февраля), и срок оплаты (28 февраля, в
  формате `28/02/25`), в README датасета ожидается именно срок оплаты.
  Мы сознательно не стали писать спецправило под один файл — эвристика
  "первая дата в тексте" универсальна и предсказуема, а неоднозначность
  двух дат в одном документе — устойчивый паттерн реальных документов,
  который правильнее помечать как "требуется ручная проверка", а не
  разрешать угадыванием.
- **`amount`: приоритет "Итого с НДС" над "Итого без НДС" и суммой самого
  НДС.** В нескольких документах фигурируют 2-3 суммы (сумма без налога,
  сам налог, сумма с налогом) — для целей казначейства релевантна именно
  сумма к оплате (с НДС).
- **`subject`/`contractor` — best-effort.** Не входят в контрольную таблицу
  датасета, реализованы по наиболее частым меткам (`Предмет:`,
  организационно-правовая форма + название в кавычках), без претензии на
  покрытие всех возможных форматов.

## 5. Как проверить

```bash
pytest -v                              # юнит-тесты (assert-примеры из задания + дополнительные)
python scripts/run_extract_report.py   # extract()/classify() по всем файлам dataset/
python scripts/run_subject_report.py   # check_subject() по dataset/subjects_test.txt
```

`run_extract_report.py` сверяется построчно с таблицей в `dataset/README.md`
(ожидаемые amount/date/inn/contractor). `run_subject_report.py` печатает
PASS/FAIL/EDGE-статус из `subjects_test.txt` рядом с фактическим результатом
`check_subject()` — расхождений с PASS/FAIL нет, EDGE-случаи получают
заметно более низкую уверенность (0.5–0.7), чем однозначные (0.7–0.95).

## 6. Примеры вызовов

```python
>>> from document_agent import extract, classify, check_subject

>>> extract("Сумма: 1 250 000,00 руб. ИНН 7701234567")
{'amount': 1250000.0, 'date': None, 'inn': '7701234567', 'contractor': None, 'subject': None}

>>> classify("Счёт на оплату №12 от 01.03.2025 ...")
('invoice', 1.0)

>>> check_subject("Поставка минеральных удобрений (карбамид марки Б)")
(True, 0.7, "предмет относится к категории 'агрохимия'")

>>> check_subject("Аренда офисного помещения, г. Краснодар")
(False, 0.85, "'аренда офиса' не относится к сельхоз-деятельности")
```

Графиков/визуализаций нет — все три функции возвращают структурированные
данные, для проверки достаточно табличных отчётов из `scripts/` (вывод см.
выше и в `RESULTS.md`).

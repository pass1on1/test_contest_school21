# RESULTS

1. **`extract()`** — regex + нормализация чисел/дат, без NLP. На 6 файлах
   датасета amount/inn/contractor совпадают с ожидаемыми на 100%; date
   совпадает на 5 из 6 — расхождение на `invoice_002.txt` (issue-дата вместо
   даты "оплата до") задокументировано в README как осознанный выбор
   универсальной эвристики "первая дата в тексте" вместо спецправила под
   один файл.
2. Главная сложность в `extract()` — выбор правильной суммы среди 2-3
   похожих кандидатов в документе (сумма без НДС / сам НДС / сумма с НДС).
   Решено скорингом по контексту вокруг найденного числа, а не первым
   попавшимся совпадением.
3. Отдельно обработан случай суммы прописью без единой цифры
   (`invoice_002.txt`, "девятьсот тысяч рублей") — написан компактный парсер
   русских числительных как fallback, когда regex по цифрам ничего не нашёл.
4. **`classify()`** — взвешенные ключевые слова с двойным весом для
   совпадений в заголовке документа (первые 120 символов), порог разрыва
   топ-1/топ-2 = 2 для отсечки в `unknown`. Все 7 файлов датасета
   классифицированы верно, включая корректный `unknown` на испорченном OCR.
   Слабое место: пороговое значение подобрано вручную на 7 примерах, для
   продакшена нужна выборка побольше и, вероятно, ML-классификатор вместо
   словаря.
5. **`check_subject()`** — только rule-based вариант (категории +
   keyword-matching), LLM-путь из задания не реализован из-за дедлайна — а
   не потому что не нужен: словарь категорий придётся регулярно расширять
   вручную, LLM с few-shot справился бы с формулировками вне словаря лучше.
   На 15 тестовых предметах: все 8 PASS и все 6 FAIL совпали с ожиданием; 3
   EDGE-кейса получили ожидаемо более низкую уверенность (0.5–0.7) вместо
   уверенных 0.7–0.95 у однозначных случаев — качественно ведёт себя
   правильно, хотя финальный bool для EDGE — по сути монетка.
6. Самое слабое место всего решения — отсутствие настоящего OCR/классификации
   на уровне символов: `scan_ocr_001.txt` обработан корректно (`unknown` +
   `None`/дата из случайно "читаемой" вставки в тексте) только потому что
   тестовый файл создан искусственно и сохранил часть текста читаемым;
   реальный OCR-мусор потребовал бы отдельного модуля коррекции символов
   перед extract/classify.
